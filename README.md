# Telegram-бот для проверки SEO-статей врачами

Этот бот работает так:

1. Врач открывает бота и вводит фамилию.
2. Бот показывает статьи, где в таблице стоит статус `На проверке у врача`.
3. Врач выбирает статью.
4. Бот показывает структуру статьи по `H2`.
5. Затем бот присылает разделы по одному.
6. На каждом разделе врач может написать комментарий.
7. При ответе с цитатой бот сохраняет и комментарий, и процитированный фрагмент.
8. На экране структуры бот может отправить ссылку на `Google Docs`.
9. В конце сценария бот может перевести врача в отдельный режим проверки иллюстраций.

## Откуда бот берёт данные

- `Google Sheets`:
  - колонка `D` — статус
  - колонка `H` — ссылка на документ
  - колонка `J` — врач
- `Google Docs`:
  - бот читает структуру документа и режет текст на разделы `H2`

## Как бот ходит в Google

Основной вариант в этом проекте:

- бот не подключается к `Google Cloud` напрямую;
- бот отправляет запрос в `Google Apps Script Web App`;
- `Apps Script` уже сам читает таблицу и документы от имени вашего Google-аккаунта.

Это удобно, потому что:

- не нужен `service account JSON`;
- не нужно бороться с запретом на создание ключей;
- настройка для неразработчика проще.

## Что лежит в проекте

- `app/bot.py` — сценарий бота
- `app/google_clients.py` — связь с Google
- `app/storage.py` — локальное хранение комментариев
- `apps_script/Code.gs` — код, который нужно вставить в Google Apps Script
- `.env.example` — пример настроек
- `Dockerfile` — готовый контейнер для деплоя на хостинг

## Настройки в `.env`

Скопируйте `.env.example` в `.env` и заполните:

- `BOT_TOKEN` — токен от `BotFather`
- `GOOGLE_ACCESS_MODE=apps_script`
- `GOOGLE_SPREADSHEET_URL` — ссылка на таблицу
- `COMMENTS_SPREADSHEET_URL` — ссылка на отдельную таблицу для комментариев
- `GOOGLE_SOURCE_SHEET_NAME` — имя листа, например `Темы МКБ`
- `GOOGLE_PENDING_STATUS_VALUE` — обычно `На проверке у врача`
- `GOOGLE_PENDING_STATUS_ALIASES` — старые варианты этого же статуса, например `Врач проверяет`
- `GOOGLE_APPROVED_STATUS_VALUE` — обычно `Проверено врачом`
- `APPS_SCRIPT_WEBAPP_URL` — адрес опубликованного Apps Script
- `APPS_SCRIPT_SECRET` — секрет, который вы сами придумаете и вставите и в `.env`, и в `Code.gs`
- `COMMENTS_SHEET_NAME` — лист для комментариев, например `Комментарии врачей`
- `EXCLUDED_SECTION_TITLES` — разделы, которые не нужно показывать врачу

Если `COMMENTS_SPREADSHEET_URL` пустой:

- комментарии пишутся в ту же таблицу, где контент-план.

Если `COMMENTS_SPREADSHEET_URL` заполнен:

- комментарии пишутся в отдельную таблицу;
- статьи при этом продолжают читаться из основной таблицы.

## Как настроить Google Apps Script

### 1. Создайте Apps Script

1. Откройте [script.google.com](https://script.google.com/)
2. Нажмите `Новый проект`
3. Удалите весь стандартный код
4. Скопируйте код из `apps_script/Code.gs`
5. Вставьте его в редактор

### 2. Придумайте секрет

В самом верху файла есть строка:

```javascript
const APP_SECRET = 'REPLACE_WITH_LONG_RANDOM_SECRET';
```

Замените `REPLACE_WITH_LONG_RANDOM_SECRET` на любой длинный секрет, например:

```text
medreview-2026-secret-92f4d7a1
```

Тот же самый секрет потом нужно вставить в `.env` в переменную:

```text
APPS_SCRIPT_SECRET=medreview-2026-secret-92f4d7a1
```

### 3. Дайте скрипту доступ к таблице и документам

Поскольку `Apps Script` запускается от вашего аккаунта, обычно отдельный доступ выдавать не нужно, если:

- таблица принадлежит вам или уже доступна вам;
- документы доступны вам.

Если какие-то документы лежат у коллег, у вашего аккаунта должен быть к ним доступ.

### 4. Опубликуйте Web App

1. Нажмите `Deploy`
2. Нажмите `New deployment`
3. Выберите тип `Web app`
4. В поле `Execute as` выберите `Me`
5. В поле доступа выберите вариант, который позволит боту обращаться по ссылке
   Обычно подходит `Anyone` или `Anyone with the link`
6. Нажмите `Deploy`
7. Скопируйте выданный `Web app URL`

Этот адрес вставьте в `.env`:

```text
APPS_SCRIPT_WEBAPP_URL=https://script.google.com/macros/s/...../exec
```

## Локальный запуск

```bash
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
python -m app.main
```

## Размещение на bothost.ru

Для `bothost.ru` проект уже подходит:

- бот работает через `long polling`
- `webhook` не нужен
- можно деплоить либо напрямую из `GitHub`, либо через `Dockerfile`

Обычно на хостинге нужно:

1. Загрузить файлы проекта
2. Настроить переменные окружения
3. Если Bothost не подхватит `Dockerfile` автоматически, указать команду запуска:

```bash
python -m app.main
```

Важно:

- на бесплатном тарифе Bothost данные стираются при перезапуске, поэтому локальная база `SQLite` не будет надёжно храниться между рестартами;
- для этого бота лучше использовать тариф с постоянным хранилищем, если вам важно сохранять резервные комментарии и регистрацию чата для итоговых отчётов.

## Если хотите использовать service account

Такой режим в проекте тоже сохранён как запасной, но для вашей ситуации он не нужен.

Для него нужно было бы использовать:

- `GOOGLE_ACCESS_MODE=service_account`
- `GOOGLE_SERVICE_ACCOUNT_FILE`
или
- `GOOGLE_SERVICE_ACCOUNT_JSON`

## Проверка

После запуска врач сможет:

- ввести фамилию;
- увидеть свои статьи;
- открыть структуру;
- пройти по разделам;
- оставить комментарии.

Комментарии:

- всегда сохраняются локально в `SQLite`;
- при настроенном Apps Script также пишутся в лист `Комментарии врачей`;
- этот лист может быть либо в основной таблице, либо в отдельной таблице через `COMMENTS_SPREADSHEET_URL`.

Комментарии с цитатой:

- врач может выделить фрагмент текста в сообщении бота;
- затем ответить на него с цитатой;
- бот сохранит и сам комментарий, и поле `Цитата`.
